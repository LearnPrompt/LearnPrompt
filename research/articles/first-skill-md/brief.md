# Brief：第一个 SKILL.md 怎么写

## 要解决的问题

很多人第一次写 Skill，会把它当成“更长的提示词”，堆很多理念，却答不出四件事：
什么时候触发、需要什么输入、按什么顺序做、怎么算完成。结果 Skill 要么不被加载，
要么一被加载就到处乱触发，或者执行完没人能判断对错。本文要把“写第一个 SKILL.md”
从抽象模板变成一次可校验、可打包的真实操作。

## 目标读者

已经在用 Claude Code 或类似 coding agent、准备把某个每天重复的流程固化成 Skill 的
实践者。默认读者会用命令行、能读 Markdown，但没系统读过官方 Agent Skills 规范。

## 学习目标

读者读完能够：

1. 判断一个流程该不该做成 Skill。
2. 说清 Skill 的三级渐进披露：哪层何时加载、各占多少上下文、由什么触发。
3. 写出一个能通过官方校验的最小 `SKILL.md`，并打包成 `.skill`。
4. 记住 `name` 与 `description` 的官方硬边界，避免第一次就踩红线。
5. 识别“再写一段提示词”补不了的结构性问题。

## 中心结论

一个能被稳定触发和执行的 Skill，价值不在“写得长”，而在于它把发现元数据、目录结构、
渐进披露、脚本和验收边界这几件事各归其位。`name` 与 `description` 会始终加载，
官方以请求是否匹配 `description` 作为读取正文的条件，所以它是主要匹配依据；两者都有官方长度和字符约束。

## 明确不做

- 不把 Skill 讲成写作技巧，重点是结构与边界。
- 不把某个第三方模板宣称为行业标准。
- 不用一次本地实验给不同 Agent 框架排名。
- 不复制橙皮书或任何第三方的图片；只把橙皮书作为中文主题地图链接。

## 需要的证明

- 官方 Agent Skills 文档对结构、字段约束、三级加载的一手说明。
- 官方 skill-creator 校验脚本对字段规则的可运行验证。
- 一个能被读者复现的最小 Skill：校验通过 + 三个反例被拒 + 打包成功。

## 验收条件

- 事实有官方一手来源，易过时信息标注核验日期。
- Showcase 能被读者用同样命令复现，原始日志脱敏冻结。
- 正文回答“为什么、怎么做、什么时候不该做”。
- 至少一张有教学作用的本地图片，许可闭环完整。
- 站点构建与确定性校验脚本通过；独立审稿在本轮之外单独进行。
