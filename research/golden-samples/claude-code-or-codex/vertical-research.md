# 纵向研究：为什么不能用“哪个更强”来选

验证日期：2026-07-10

## 1. 工具不是模型

一次 coding-agent 任务的结果至少由四层共同决定：模型、Harness、仓库可读性、任务与验收。工具名称只能指向其中一部分。同一个工具切换模型、权限、沙箱或项目规则后，表现可能明显不同。

因此，本文现场实验固定了仓库和提示词，却无法固定模型、缓存、服务负载和内部策略。它适合展示工作流和故障，不适合宣称速度或智力排名。

## 2. “本地 vs 云端”已经不够

Claude Code 官方文档列出终端、IDE、桌面和浏览器入口，并提供 background agents、agent teams 与 worktree。Codex 同时有本地 CLI、桌面应用和 cloud tasks。两者不是单一产品面。

更稳定的判断维度是：

- 你是否需要在任务进行中频繁纠偏？
- 任务是否依赖本机服务、登录态或未提交文件？
- 输入和验收是否足够明确，可以交给后台环境？
- 失败会影响哪些文件、账号或外部系统？
- 结果能否由测试、构建或第二个 reviewer 独立判断？

## 3. 权限是选择的一部分

本地交互的优势是反馈快，但它也更接近真实文件和凭据。Claude Code 提供 permission rules 与多种 mode；Codex CLI 提供 approval policy 和 sandbox。名称不同，工程问题相同：默认最小权限，高风险动作单独批准，并把验证命令写在任务开始前。

## 4. 现场失败比成功更有教学价值

本次 Codex 首次执行没有进入任务，而是因为本地配置默认模型 `gpt-5.6-sol` 与已安装 CLI 版本不兼容而停止。切换到本机缓存中兼容的 `gpt-5.4` 后成功。

这说明“选择 Codex”还不够：版本、模型配置、沙箱和仓库状态共同组成实际执行环境。文章应保留这种失败，而不是只截取成功输出。

## 5. 双工具协作的正确分界

双工具不是让两个 Agent 同时改同一组文件。更可靠的结构是把角色拆开：一个负责实现，另一个只读审查；或者一个做长上下文探索，另一个接收结构化 handoff 执行。共享的不是模糊聊天记录，而是任务、边界、验收命令和证据。
