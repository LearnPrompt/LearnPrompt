# 纵向研究：为什么换模型常常治不好 Agent

验证日期：2026-07-10

## 1. 模型只负责提出下一步

真实任务还需要文件、命令、权限、状态与验收。缺少这些，模型即使知道“应该做什么”，也可能找不到正确文件、无法判断命令是否安全，或在没有证据时宣布完成。

因此可以把 Agent 的可靠性粗略写成：模型能力 × 环境可读性 × 反馈质量 × 约束执行。这个乘法是解释模型，不是可计算公式；它强调任何一项接近零都会拖垮整体。

## 2. 五组件分别防什么失败

- 指令防止工作方式含糊，例如不知道先读规则还是直接修改。
- 能力防止“口头允许但技术上做不到”，也防止无必要地开放网络或 shell。
- 约束把风险边界从提醒变成拒绝、审批或沙箱。
- 状态让任务跨步骤延续，并保存失败证据而不是只存结论。
- 编排把探索、实现、验证和停止条件连接起来，避免 worker 自己给自己打分。

## 3. 为什么长说明书不是答案

OpenAI 的 Harness 工程文章主张给 Agent 一张地图，而不是一份上千页手册。理由很直接：入口文档越长、规则越重复，越容易互相矛盾和过时。关键边界应由 lint、测试、类型、权限和结构检查执行；文档负责解释意图与导航。

## 4. 为什么 evaluator 要独立

Anthropic 的长任务研究指出，自我评价可能产生偏差。把 generator 与 evaluator 分开，验收条件写进结构化 artifact，可以降低“代码写完了所以测试应该通过”的自证循环。这里的独立既可以是另一模型，也可以是确定性测试、静态检查或人类 reviewer。

## 5. 最小 Harness 的边界

Showcase 只验证五个部件存在且字段完整。JSON 文件本身不会阻止真实命令执行；生产系统仍需要把约束接到沙箱、权限层、CI 和审计日志。它的价值是让缺失组件一眼可见，不是假装安全已经解决。
