# 纵向研究：一条问题追到底

中心问题：**同一个任务，怎么决定哪些动作能自动跑、哪些必须人工批准，并且这个边界靠什么强制？**

验证日期：2026-07-11。标注：事实=有来源，综合=编辑推断。

## 表层现象

很多人用一句"我信任这个模型"来放权，或者反过来什么都手动确认。前者把一次不可逆的 push 交给概率模型，后者让 agent 退化成慢速自动补全。现象背后的真问题是：放权的单位选错了——应该按动作的风险，而不是按模型的名气。

## 机制一：风险由动作的可逆性与影响半径决定（综合）

一个动作值不值得自动跑，看两件事：能不能撤销，影响半径有多大。

- 只读（读文件、grep、跑测试查询）：零写入，随时可停，属于只读计划档。
- 工作区内写入（改源码、跑测试）：可用版本控制撤销，影响半径在仓库内，属于普通自动执行档。
- 越界动作（push、部署、删库、改密钥、联网、写工作区外）：不可逆或触及外部系统，属于高风险档，必须人工批准并配回滚。

这三档是工具无关的判断框架。难点在于：判断框架不等于强制机制。光在提示词里写"请小心"不构成边界。

## 机制二：Claude Code 用两条轴强制（事实）

Claude Code 把"能不能做"拆成两层，且都由 Claude Code 进程强制，不是模型自觉：

1. 权限模式决定整体姿态。`plan` 只读探索、不改源文件；`acceptEdits` 自动接受工作目录内的编辑与 mkdir/mv 等；`default`（Manual）逐个提示；`bypassPermissions` 跳过提示（仅限隔离环境）。
2. allow/ask/deny 规则决定单个工具调用。求值顺序固定为 deny → ask → allow，第一个命中即决定，规则特异性不改变顺序；任一层级的 deny 都压过任何 allow。所以 `deny Bash(git push *)` 能在整体放开 Bash 的前提下单独锁死 push。

官方明确写道：规则由 Claude Code 强制，提示词或 CLAUDE.md 只影响模型想做什么，不改变 Claude Code 允许什么。这就是"边界不靠信任模型"的一手依据。

## 机制三：Codex 用另一套两条轴强制（事实）

Codex 的两条轴名字和语义都不同：

1. 审批策略（approval_policy）决定模型何时向人请求批准：`untrusted` 只自动跑受信命令、遇到非受信命令升级给用户；`on-request` 由模型临场决定何时问；`never` 从不问、失败直接回给模型；`on-failure` 已废弃。
2. 沙箱模式（sandbox_mode）决定 OS 级的文件与网络边界：`read-only`、`workspace-write`、`danger-full-access`。workspace-write 下出站网络默认关闭。

两条轴正交：沙箱控制命令运行后的资源边界，审批策略控制运行前什么时候可能询问人。`on-request` 由模型决定何时问，因此不能单独承担“高风险动作必问”的确定性边界；高风险能力应保持关闭，由任务外的人类 gate 显式重新授权。

## 机制四：为什么两套术语不能互译（综合 + 事实）

plan 模式和 read-only 沙箱都表现为只读，但强制层不同：

- Claude Code 的 plan 是 agent 行为模式，由应用进程约束模型的工具调用。
- Codex 的 read-only 是内核沙箱，由操作系统拒绝写系统调用。

同理，Claude Code 的 allow/ask/deny 是预先写死的确定性规则；Codex 的 `on-request` 是让模型临场判断何时问人，依赖模型。把"plan 模式=read-only 沙箱"或"allow 规则=approval policy"直接画等号，会让读者在一个产品里用错原语。

## Showcase 追问：边界真的由内核强制吗（事实）

用 `codex sandbox` 对同一批动作逐条裁决（见 showcase/experiment-output.md）：

- read-only 下 `cat` 成功、`echo >>` 被拒（Operation not permitted）。
- workspace-write 下工作区写入成功，但写 `$HOME` 仍被拒、出站网络仍被拒。

拒绝来自内核返回的 `Operation not permitted`，不是模型的礼貌。这直接证明：普通自动执行与高风险动作之间的线，可以由沙箱机械划定。

## 失败模式与边界条件

- 用模型名字放权：把不可逆动作交给概率判断。修法：按动作风险放权。
- 只在提示词里写"别 push"：不构成边界，模型仍可能执行。修法：用 deny 规则或沙箱强制。
- 把 workspace-write 当成安全：它只挡工作区外和网络，工作区内的破坏（如覆盖源文件、跑 `git reset --hard`）仍会发生。修法：高风险的工作区内动作也要单独 ask/deny 或人工在场。
- 平台差异：本文只实测 macOS Seatbelt；其他平台的实现、可用性与报错未核验，结论必须标平台与版本。
- bypassPermissions / danger-full-access：只应在隔离容器里用，且这类模式下仍有 rm -rf / 之类的断路器提示，但不能替代真正的隔离。

## 迁移到读者自己的任务（综合）

读者应当能对任一任务做三件事：先写清楚成功与验收；再把动作按可逆性和影响半径分三档；最后在自己用的工具里选对应原语，把高风险动作钉在人工批准这一侧，并为其准备回滚。
