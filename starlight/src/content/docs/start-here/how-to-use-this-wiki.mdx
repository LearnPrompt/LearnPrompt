## 不要顺序读，先写下当前阻塞

打开目录前，先用一句话写清楚：“我正在做什么，卡在哪一步，完成的证据是什么？”

例如，“我想学 Codex”还不能决定读哪篇；“我有一个本地仓库，想让 Codex 改一处文档链接，完成后测试通过且不允许 push”就能直接进入 [Codex CLI 入门工作流](/codex/codex-cli-workflow/) 和 [沙箱与审批](/codex/sandbox-and-permissions/)。

## 三种读法

### 第一次做项目：沿闭环读

先读 [AI 实战学习地图](/start-here/ai-practice-map/)，再用一个可丢弃或容易回滚的小项目跑完“任务 → 计划 → 修改 → 验收 → 复盘”。第一次不要同时更换模型、编辑器、部署平台和项目结构。

### 正在解决问题：沿阻塞读

只打开当前决策所需的 1–3 篇：

- 不知道怎样描述任务：读 [任务、上下文与验收](/start-here/context-task-verification/)。
- 不知道选 Claude Code 还是 Codex：读 [工具选择](/ai-coding/choose-claude-code-or-codex/)。
- Agent 会做但不可靠：从 [Harness 五组件](/agent-engineering/what-is-harness/) 找缺失层。
- 重复流程越来越长：先判断 [是否值得做成 Skill](/agent-skills/what-are-agent-skills/)。
- 跨会话接不上：区分 [记忆层](/agent-engineering/memory-layer/) 与 [可交接 Markdown](/obsidian-ai/markdown-as-agent-memory/)。

### 项目做完了：沿失败读

拿真实失败反查教程，而不是凭印象写规则。失败如果来自边界不清，回到 instruction / constraint；来自结果难判断，回到 feedback；来自多步骤状态漂移，回到 orchestration；只有可重复且稳定的流程，才考虑固化成 Skill。

## 怎样读一篇深度教程

一篇教程可以分四遍读：

1. **先看任务与边界**：确认它解决的是你的问题，也确认“什么时候不要用”。
2. **再看图和 Showcase**：理解正常路径与拒绝路径，别只复制最终命令。
3. **动手重放**：把合成输入换成自己的小样本，保留命令、退出码和 diff。
4. **最后读来源**：易变的命令、权限、价格和版本必须回官方资料复核。

Showcase 证明的是一个冻结合同在给定环境中的行为，不自动证明所有项目、所有版本或所有权限组合都成立。

## 认识编辑流程状态

下面四项是生产计划和 `research/` 使用的编辑控制面状态，不是四种都能写入公开页面 frontmatter 的展示标签。公开深度教程只有在全部门禁通过后才标成 `verified`；未完成状态留在控制面中，不伪装成公开成品。

| 状态 | 代表什么 | 你应该怎样使用 |
| --- | --- | --- |
| `verified` | 研究、Showcase、独立评审、教学图和机械门禁均通过 | 可以作为当前实践起点，易变事实仍回官方核对 |
| `review` | 正在独立评审，结论尚未冻结 | 只作草稿参考，不据此做高风险决策 |
| `partial` | 研究或证据链不完整 | 不把示例当成已验证方案 |
| `blocked` | 在限定重试预算内仍有明确阻塞 | 先看阻塞记录，不重复宣称成功 |

导航页没有 Showcase 状态，也不会为了凑数套用 `verified`。它们的职责是让你找到已经通过门禁的正文。

## 留下一张学习 receipt

每读完一篇，只保存下面六项：

```md
# 学习 receipt
- 当前任务：
- 使用的教程：
- 我实际运行的输入或命令：
- 通过的证据：
- 失败或不适用之处：
- 下一次要写回的规则、记忆或 Skill：
```

这张 receipt 比“读完打卡”更有用：下一位 Agent 能看到你做过什么、什么没有证明，以及下一步从哪里继续。

## 来源与许可怎样看

文章底部保留真实使用过的来源，不再显示独立来源卡片组件。来源分三层：

- **一手资料**：产品官方文档、规范、代码仓库和发布记录，用来核对能力与版本。
- **二手主题地图**：例如橙皮书，用来发现问题空间，不替代官方事实。
- **现场实验**：仓库内冻结 fixture、脚本、receipt 和 review，用来证明文章自己的 Showcase。

完整栏目映射、Orange Book 署名和许可边界见 [来源索引](/sources/source-index/)。
