## 这不是课程目录，而是入口判断器

不要先问“我该把所有文章按什么顺序读完”，先问三个更有用的问题：

1. 我现在有没有一个真实项目或可丢弃的练习仓库？
2. 当前阻塞发生在任务描述、工具执行、结果验收，还是跨会话持续？
3. 七天后我希望留下什么可检查的产物？

LearnPrompt 目前有八条实践路径。它们按问题层级组织，不按工具热度排序。

## 八条路径各解决什么

| 路径 | 它解决的问题 | 建议第一篇 | 路径完成的证据 |
| --- | --- | --- | --- |
| AI 编程 | 把模糊想法变成可执行、可验收的小切片 | [最小工作流](/ai-coding/minimum-agentic-coding-workflow/) | 一次 plan → patch → verify → learn 闭环 |
| Claude Code | 在本地项目中管理指令、会话和扩展 | [安装与第一个项目](/claude-code/install-and-first-project/) | 一个可回滚的小仓库改动和真实检查结果 |
| Codex | 选择 CLI、IDE、App、Cloud，并管理沙箱和 review | [四个执行面](/codex/codex-form-factors/) | 一张执行面选择和一个可审查的任务 receipt |
| Agent 工程 | 诊断 Agent 为什么会跑偏、越界、失忆或停不下来 | [Harness 五组件](/agent-engineering/what-is-harness/) | 指令、能力、约束、状态、编排五层缺口表 |
| Agent Skills | 把稳定重复流程变成可触发、可测试的工作包 | [Skill 是什么](/agent-skills/what-are-agent-skills/) | 一个通过触发与结构检查的 `SKILL.md` |
| Loop Engineering | 让一次任务产生下一轮可验证动作 | [Loop 的五个动作](/loop-engineering/five-moves/) | 带状态、验收、持久化和停止条件的最小循环 |
| Obsidian AI | 让资料、项目状态和经验可放置、可路由、可交接 | [Vault 目录 contract](/obsidian-ai/vault-directory-for-ai/) | 一份 placement contract 或 handoff packet |
| Hermes / OpenClaw | 理解长驻 Agent 的学习写入、消息路由、部署和预算 | [Hermes 学习循环](/agent-frameworks/hermes-learning-loop/) 或 [OpenClaw 架构](/agent-frameworks/openclaw-architecture-guide/) | 一张写入审批表或可审计的 Gateway 路由图 |

## 先按失败模式定位

如果你已经有项目，下面这张表通常比“新手/高手”标签更快：

| 你看到的失败 | 先读 | 暂时不要做 |
| --- | --- | --- |
| Agent 一上来就改错地方 | [任务、上下文与验收](/start-here/context-task-verification/) + [指令层](/agent-engineering/instruction-layer/) | 不要先增加更长的万能 prompt |
| 不知道该不该开自动执行 | [Plan、Auto 与人工审批](/ai-coding/plan-auto-approval-boundary/) | 不要按“信不信模型”划权限 |
| Claude Code / Codex 二选一反复纠结 | [工具选择](/ai-coding/choose-claude-code-or-codex/) | 不要只比模型榜单 |
| 修改看似完成，却没人能证明正确 | [反馈层](/agent-engineering/feedback-loop/) | 不要把“已完成”当验收证据 |
| 长任务中途漂移 | [Claude Code 长任务模式](/claude-code/advanced-conversation-patterns/) 或 [编排层](/agent-engineering/orchestration-layer/) | 不要无限延长同一段会话 |
| 规则越来越多、每次都全量加载 | [最小 CLAUDE.md](/claude-code/minimum-claude-md/) + [Skill 触发边界](/agent-skills/trigger-rules-and-structure/) | 不要把所有知识塞进 always-on 指令 |
| 新 Agent 接手后只能重新调查 | [可交接 Markdown](/obsidian-ai/markdown-as-agent-memory/) | 不要只粘贴聊天记录 |
| 知识库分类漂亮但 Agent 经常放错 | [Vault 放置 contract](/obsidian-ai/vault-directory-for-ai/) | 不要继续增加含义重叠的文件夹 |
| 长驻 Agent “会学习”但写入不可控 | [Hermes 学习循环](/agent-frameworks/hermes-learning-loop/) | 不要把 memory、Skill 和模型训练混为一谈 |
| OpenClaw 配好了却消息不通或暴露过大 | [架构导读](/agent-frameworks/openclaw-architecture-guide/) + [上线决策](/agent-frameworks/deployment-channels-cost/) | 不要用“进程在运行”代替端到端健康检查 |

## 三条完整读者路线

### 路线 A：第一次使用 AI 编程

1. [任务、上下文与验收](/start-here/context-task-verification/)
2. [AI 编程项目清单](/ai-coding/project-checklist/)
3. [Agentic Coding 最小工作流](/ai-coding/minimum-agentic-coding-workflow/)
4. [Claude Code 与 Codex 怎么选](/ai-coding/choose-claude-code-or-codex/)
5. 任选一个工具完成一次真实小改动

出口不是“了解 AI 编程”，而是一个干净 diff、一条通过的验收命令和一条写回项目的经验。

### 路线 B：已经使用 Claude Code 或 Codex

先按当前工具分支，不要把另一条产品线当必修：

- **Claude Code 支线**：[最小 CLAUDE.md](/claude-code/minimum-claude-md/) → [长任务上下文](/claude-code/advanced-conversation-patterns/) → [Skill、Hook、MCP 怎么选](/claude-code/skills-hooks-mcp-roles/)。
- **Codex 支线**：[Codex CLI 工作流](/codex/codex-cli-workflow/) → [沙箱与审批](/codex/sandbox-and-permissions/) → 按任务选择 [Cloud 适配](/codex/codex-cloud-task-fit/) 或 [auto-review 边界](/codex/auto-review-boundaries/)。
- **两条线共同补课**：[Plan、Auto 与人工审批](/ai-coding/plan-auto-approval-boundary/) → [反馈层](/agent-engineering/feedback-loop/)。
- **只有确实跨工具协作时**，再读 [双线 Git 交接](/codex/codex-claude-code-dual-track/)，不要为了“工具齐全”强行增加一次 handoff。

出口是一套与当前工具匹配、能被另一位 Agent 或 reviewer 重放的规则、diff 和检查结果；双工具用户再额外留下 handoff contract。

### 路线 C：建设长期 Agent 工作流

1. [Harness 五组件](/agent-engineering/what-is-harness/)
2. 依次审计 [指令](/agent-engineering/instruction-layer/)、[约束](/agent-engineering/constraint-layer/)、[记忆](/agent-engineering/memory-layer/)、[反馈](/agent-engineering/feedback-loop/) 与 [编排](/agent-engineering/orchestration-layer/)
3. [Loop 的五个动作](/loop-engineering/five-moves/)
4. 判断重复流程是否值得升级成 [Skill](/agent-skills/what-are-agent-skills/)
5. 用 [可交接 Markdown](/obsidian-ai/markdown-as-agent-memory/) 或框架自己的审批机制保存可观察状态

出口是一台有输入合同、状态、停止条件、验收和持久化边界的最小循环，而不是一个“永远自动运行”的口号。

## 7 天实践路线

| 天 | 动作 | 当天必须留下的证据 |
| --- | --- | --- |
| 1 | 选一个一小时内可完成的小任务 | 目标、范围、禁区、验收 |
| 2 | 让 Agent 只读探索并给 3–6 步计划 | 文件清单、风险和未确认项 |
| 3 | 只实现一个最小切片 | 可读 diff，不包含顺手重构 |
| 4 | 跑最快的相关检查 | 原始命令、退出码、人工检查点 |
| 5 | 主动构造一个失败或边界反例 | 被拒绝的输入和原因 |
| 6 | 把一条可复用教训写回项目 | 一条短规则、记忆或 checklist |
| 7 | 用 fresh 会话或另一位 reviewer 重放 | 能否在不依赖聊天记忆时继续 |

## 任务卡模板

```md
# 目标
完成后，用户或系统能观察到什么变化？

# 上下文
项目、当前状态、关键入口和已知证据是什么？

# 范围与禁区
允许改什么？不允许改什么？哪些动作必须先问？

# 验收
运行什么命令？人工看什么？失败时输出什么？

# 交付
需要 diff、receipt、review、PR 草稿，还是下一轮 handoff？
```

模板写不清时，不要急着选更强模型；先回到 [任务、上下文与验收](/start-here/context-task-verification/)。

## 如何判断真的学会了

读完一篇教程后，用四个问题验收：

1. 我能否把示例换成自己的小输入，而不是照抄最终答案？
2. 我能否说出正常路径之外至少一个应该拒绝的边界？
3. 我是否保留了命令、退出码、diff 或人工检查的原始证据？
4. fresh Agent 能否只看我的产物继续，而不依赖这次聊天？

四项中有一项答不上来，就回到对应教程的 Showcase 或练习，不要继续扩工具栈。
