# Showcase：一台最小编排状态机

这个 Showcase 用一个无网络、无第三方依赖的最小状态机回答一个问题：编排层到底在决定什么。同一个任务经历 `INSPECT → IMPLEMENT → VERIFY → RETRY → STOP/ESCALATE`，我们保存它的每一次状态转换、一次真实失败、修正后通过、重试预算，以及预算用尽后的升级出口。

它证明的不是某个模型更聪明，而是：把执行角色（worker）、判定角色（evaluator）和控制流（orchestrator）分开之后，顺序、路由、预算、停止与升级都变成可观察、可复现的确定性动作，而不是靠感觉。

## 诚实边界

worker 是一个**确定性桩**（deterministic stub），按顺序吐出预先准备好的候选实现，用来代表“执行角色每一轮交出的产物”。它不是在线大模型，本 Showcase 也没有运行任何在线模型，因此两个场景**不构成任何模型排名**。这里刻意演示的是编排层的控制流，不是模型智能。

## 目录

```text
state-machine/
├── acceptance.mjs     # 验收契约：三条 输入→期望 用例（先定义 done）
├── worker.mjs         # 执行角色：确定性候选生成器（非 LLM）
├── evaluator.mjs      # 判定角色：独立跑验收，只给 pass/fail + 定位
├── orchestrator.mjs   # 编排层：状态机 + 路由 + 重试预算 + 停止/升级
├── run.mjs            # 跑两个场景并打印状态转换日志
└── result.txt         # 冻结的真实运行输出（已脱敏）
```

三个角色严格分工：worker 只产出、不判分；evaluator 只判分、不改码；orchestrator 只读判定后决定去哪一步，既不写码也不判分。做事的和判分的不是同一个角色——这样“我觉得写完了”不能替代一次真实验收。

## 环境

- macOS，Node v24.11.0，无网络、无第三方依赖。
- 所有命令在本目录内以相对路径运行。

## 复现步骤

```bash
node run.mjs        # 依次跑场景 A、B，打印每一次状态转换，最后返回整体退出码
```

## 2026-07-11 的真实结果

- 场景 A（`retry_budget=2`）：第一版候选漏掉小写化，VERIFY 得到 `expected=hello-world actual=Hello-World`，被路由回 IMPLEMENT；第二版补上小写化，VERIFY 三条全过，走 `STOP_DONE`，退出码 0。用掉 1 次重试，预算内收敛。
- 场景 B（`retry_budget=1`，worker 卡住）：两版候选都漏掉小写化，第二次 VERIFY 仍失败且预算已用尽，状态机走 `ESCALATE`，`handoff=human`，退出码 1。
- `run.mjs` 整体退出码 EXIT=2：只要有场景以升级收尾，整体判为“需要人接手”，用退出码把这个事实暴露给上游脚本。

完整脱敏输出见 `result.txt`。

## 它证明了什么

- 编排层的五个决定（顺序、角色路由、重试预算、停止、升级）都能落到可观察、可复现的状态转换上。
- 停止（STOP_DONE）和升级（ESCALATE）是两条真实分支：场景 A 触发前者，场景 B 触发后者，都不是文档里的装饰句。
- 重试预算是一个真实的整数上限：`retry_budget` 用尽后不再回 IMPLEMENT，而是换出口，避免死循环。
- 验收独立于执行：worker 说不出“我通过了”，只有 evaluator 跑完 acceptance 才产生 pass/fail。

## 它没有证明什么

- worker 是确定性桩，不是在线模型；本 Showcase 不比较模型能力，也不构成模型排名。
- 一次收敛不代表真实任务长期可靠；真实编排还要接沙箱、CI、审计日志与人工审批。
- 状态机能停能升级，不代表候选内容设计得对；把它接到真实工具后，还需要能力层与约束层兜底。
- 这里的“升级给人”只是打印一行 handoff；真实系统要把上下文、失败证据和续跑入口一并交接。
