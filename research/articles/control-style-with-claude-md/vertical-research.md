# 纵向研究：从一次风格失败追到放置判断与维护循环

中心问题：当规则已经开始膨胀时，为什么继续给根级 `CLAUDE.md` 加段落通常不是最优解？应该如何把失败拆到不同层级？

## 表层现象

项目最初只有一份很短的 `CLAUDE.md`：真实命令、验收命令、禁区路径、收尾格式。一切都还能工作。

但当项目同时出现 UI、API、测试、文档等不同子系统后，团队常见做法是继续把所有规则塞回根文件：

- UI 用设计 token。
- API 返回统一 envelope。
- 文档标题用某种写法。
- 测试文件遵循某种命名。

于是根文件从“每次都该知道的事实”逐渐变成“全项目规则回收站”。

## 为什么会失效

Claude Code 官方把 `CLAUDE.md` 描述成每轮都会读入的 context。换句话说，根文件里的每一行都在和本轮真实任务竞争注意力。当一条规则只对某个目录有效，却被放进全局入口时，你在每轮会话里都引入了无关上下文。

这带来两个结果：

1. 上下文噪音增加：前端任务要背 API 细则，后端任务要背 UI 细则。
2. 冲突更难发现：局部规则与全局规则混在一起时，很难看见哪一条真的互相打架。

官方文档对这两个问题分别给出提示：

- 规则要具体、简洁，过长会降低遵循。
- 如果两条规则冲突，Claude 可能任意选择一条。

这正说明“继续加更多句子”不是默认解法。

## 为什么要拆成三层

从一次失败反推，真正需要回答的是三个不同问题：

### 问题一：这是不是每次都该知道？

如果答案是“是”，它才适合进根级 `CLAUDE.md`。例如真实命令、主目录边界、统一交付格式、如何验证改动。

### 问题二：这是不是只在某些文件上成立？

如果答案是“是”，它更适合做 path-scoped rule。因为你要解决的是“加载时机”，不是“文档怎么分类”。只有当 Claude 真在处理 `src/ui/` 时，UI 规则才应该进入上下文。

### 问题三：这是不是可以被脚本机械判定？

如果答案仍是“是”，再进一步，应把它下沉到确定性检查。因为你已经不需要模型来“理解一条提醒”，而需要一个稳定裁判来判断 pass/fail。

所以三层拆法不是形式主义，而是从三个不同问题倒推出的最小结构。

## UI/API showcase 为什么是合适的教学对象

这个案例之所以好，是因为它天然包含“只对局部成立”的规则。

- UI 的 `no hard-coded hex colors` 并不是整个仓库都要关心的重点。
- API 的 `return jsonOk/jsonError` 也不应该在 UI 任务里常驻出现。

只要把两条都塞进根文件，就会立刻遇到“全局上下文被局部规则污染”的问题。反过来，只要把它们放进 path-scoped rules，再加一个脚本做机械检查，就能非常清楚地展示三层分工：

- 根文件负责告诉 Claude：局部风格在哪里、验证命令是什么。
- 作用域规则负责告诉 Claude：此刻你在 UI 还是 API。
- 检查器负责最终判定当前代码是否真的通过。

## 为什么脚本不能伪装成模型服从测试

这是本文最需要保持诚实的地方。

静态脚本能证明的，是“当前代码是否满足某条机械约束”。例如：

- 文件里是否还有十六进制颜色。
- API 处理器是否直接返回裸对象。
- 某条规则有没有正确声明 `paths`。

它不能证明的，是：

- Claude 是否确实因为读了那条 rule 才做出正确实现。
- 换一个任务、换一个上下文时是否仍会稳定遵守。
- 某条提醒比另一条提醒“更聪明”。

因此本文只把脚本当作规则分层的确定性证据，不把它包装成模型能力实验。由于本轮任务还明确受限于认证失败、不能重走原有在线链路，这个边界必须写入正文与 showcase README。

## 维护循环为什么必须包含“修剪”

很多团队会做“添加”和“验证”，却漏掉“修剪”。结果就是：

- 同一条要求既写在根文件，又写在 path rule，又写在 PR 模板里。
- 某条原本不能脚本化的规则，后来已经可以做 lint，但根文件里仍保留整段说明。
- 一个目录被重构后，旧路径规则继续留着，实际再也不会命中。

修剪的意义有两个：

1. 让根文件重新回到“每次都该知道”的体积。
2. 降低冲突概率，避免模型在旧规则和新规则之间任意选边。

如果说第一份 `CLAUDE.md` 的核心是“把重复失败写下来”，那进阶阶段的核心就是“把已经不该常驻的东西移走”。

## 编辑综合（非官方术语，需显式标注）

- “根规则 / 作用域规则 / 机械检查”是 LearnPrompt 对官方 memory、best practices、features overview 的操作化整理，不是 Claude 官方命名。
- “添加 -> 验证 -> 修剪”是对官方“当它错第二次就写进 CLAUDE.md”“文件过长要删”“给它 pass/fail 检查”的编辑综合。
- “UI 规则不应作用于 API”在本文中通过 path-scoped showcase 和脚本轨迹证明为教学结论，不是官方原句。
