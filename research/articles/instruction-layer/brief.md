# Brief：指令层——把意图变成 Agent 可执行的项目地图

## 要解决的问题

很多人以为写 Agent 的指令就是把提示词写得更长、更客气：加一句“你是资深工程师”“请认真、保持高质量”。结果 Agent 要么不读规则就动手，要么把范围越改越大，要么自己宣布“做完了”而没人能判断对错。真正的问题不在措辞，而在于这份指令有没有被翻译成一张能落到真实文件、命令和判定上的项目地图：目标是什么、能动哪些文件、按什么顺序、用什么验收、维度打架时听谁的。本文要把“写指令”从堆形容词变成一次可校验的翻译工作。

## 目标读者

已经在用 Claude Code、Codex 或类似 coding agent，准备为一个仓库写 CLAUDE.md / AGENTS.md，或给单次任务写 brief 的实践者。默认读者会用命令行、能读 Markdown，读过或写过至少一版粗糙的指令文件。

## 学习目标

读者读完能够：

1. 说清指令层的五个可执行维度：目标、范围、顺序、验收、冲突优先级。
2. 把每个维度落到一个可定位的真实资源，而不是形容词。
3. 用一份确定性检查器判断自己的指令能否被解析成可执行地图。
4. 说清指令层与能力、约束、状态、编排层的分工：指令解释意图，约束才是硬闸。
5. 按固定优先级解指令冲突，避免模糊目标和隐藏验收互相打架导致跑偏。
6. 核对 Claude Code CLAUDE.md 与 Codex AGENTS.md 的当前一手事实（加载、精确性、冲突、优先级）。

## 中心结论

指令层的价值不在长度，而在于它把意图翻译成一张 Agent 能执行、能验收的项目地图。官方文档对此有明确一手支撑：CLAUDE.md 被当作上下文而非强制配置，越具体、越简洁越被稳定遵循，两条规则互相矛盾时模型可能任选其一；Codex 的 AGENTS.md 按根到叶拼接、越近越优先，并有 32 KiB 上限。因此指令层要做的是把每个维度落到真实资源，并显式声明冲突优先级；真正的强制（拒绝、沙箱）属于约束层，用钩子而非措辞实现。

## 明确不做

- 不把指令层讲成写作技巧，重点是可执行维度与分工。
- 不把某个模板宣称为行业标准。
- 不用一次本地实验给 Claude Code、Codex 或任何模型排名。
- 不复制 harness-engineering 橙皮书或任何第三方图片；橙皮书只作为中文主题地图链接。

## 需要的证明

- Claude Code 官方 memory 文档对 CLAUDE.md 加载顺序、精确性建议、冲突处理、“上下文而非强制配置”的一手说明。
- Codex 官方 AGENTS.md 文档对发现顺序、根到叶拼接、越近越优先、32 KiB 上限的一手说明。
- agents.md 官方站点对格式开放性、就近生效、用户 prompt 覆盖一切的一手说明。
- 一个能被读者复现的最小仓库：确定性检查器给模糊/可执行指令打分（0/5 vs 5/5），同一份验收在模糊指令导向的实现上挂、在可执行指令导向的实现上过。

## 验收条件

- 核心事实有官方一手来源，易过时信息标注 2026-07-11 核验日期。
- Showcase 能被读者用同样命令复现，原始日志脱敏冻结，无绝对路径。
- 正文回答“为什么、怎么做、什么时候不该用指令层解决”。
- 至少一张有教学作用的本地图片（指令地图 + 冲突优先级），许可闭环完整。
- 站点构建与确定性校验脚本通过；独立审稿在 writer 轮之外单独进行。
