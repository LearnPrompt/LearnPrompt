# 纵向研究：为什么一份写清楚的开工清单能改变结果

中心问题：为什么同一个模型，给它开工清单就稳，丢一句“帮我改”就反复失败？
从表层现象追到机制、失败模式、权衡与边界。标注编辑综合。

## 表层现象

- 现象 A：Agent 直接在主分支上改，改坏了难以隔离与回退。
- 现象 B：Agent 猜运行命令，跑错脚本、反复试错、浪费轮次。
- 现象 C：Agent 顺手改了不该改的文件，范围不断漂移。
- 现象 D：任务“看起来完成了”，但没有客观信号证明，返工。

## 机制一：模型拿到的是上下文，不是环境

CLAUDE.md 文档明确：这些指令是 context 而非 enforced configuration。
也就是说，模型只能依据你给的上下文推断“该怎么做”。开工清单的作用，是把编辑者脑子里的
隐性知识（在哪跑、别碰哪、怎么算完成）显式写出来，变成模型每次都能读到的上下文。
清单缺一格，模型就只能猜那一格——猜错就产生上面的失败。

## 机制二：具体、可执行的指令比形容词更可靠

AGENTS.md 和 CLAUDE.md 都强调指令要具体到可验证：用 `npm test` 而不是“测试你的改动”。
原因很朴素：可执行命令能被真正运行并产生退出码，形容词不能。
因此开工清单里最有价值的不是“注意质量”，而是一条能跑的验收命令。
这也是本文 Showcase 的判据：验收字段必须像命令，散文式的“手动看一下”会被拦截。

## 机制三：真正的边界靠工具强制，不靠措辞

权限文档写得很直接：`Permission rules are enforced by Claude Code, not by the model`。
清单里写“不要碰 .env”只是软提醒；要变成硬边界，得落成 `Read(.env)` 这类 deny 规则或沙箱。
所以开工清单有两层：一层是给人和模型读的说明（AGENTS.md/CLAUDE.md），
一层是给执行器强制的配置（deny 规则、审批、沙箱）。清单负责表达意图，工具负责拒绝。

## 失败模式与对应字段（编辑综合）

| 缺失字段 | 触发的失败 | 一手依据 |
| --- | --- | --- |
| 仓库与分支 | 无隔离，改坏难回退 | git 分支与 revert 语义 |
| 运行与最快检查 | 猜命令、反复试错 | AGENTS.md 建议写构建/测试命令 |
| 禁改目录 | 范围漂移 | 权限 deny 规则按路径限制 |
| 密钥方式 | 凭据入库或泄露 | 权限 Read(.env) deny、CLAUDE.md 不放密钥 |
| 验收路径 | 只问“完成没”，无信号 | AGENTS.md 要求可执行检查 |
| 回滚方式 | 无法安全退回 | git revert 保留历史地撤销 |

## 权衡

- 清单太短：省事，但把风险留给运行时，失败半径大。
- 清单太长：接近“上千页说明书”，规则易冲突、过时；CLAUDE.md 建议单文件控制在 200 行内。
- 折中：入口像地图不像百科；一次写好六格，细节留给靠近代码的文档与测试。

## 边界条件（什么时候这套不适用或不够）

- 一次性、零风险的小实验，写清单的成本可能高于收益。
- 清单只是提醒层；高风险动作（发布、删云数据、发消息）必须叠加工具强制与人工审批。
- 清单校验只保证“写全了”，不保证“写对了”；内容正确仍需人复核与真实验收。

## 编辑综合结论

开工清单不是模板崇拜，而是把“模型只能读上下文”“具体命令可验证”“边界靠工具强制”
这三条机制固化成一份可复用、可校验的工件。它把失败从运行时提前暴露到开工前。
