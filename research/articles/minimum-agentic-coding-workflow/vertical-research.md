# 纵向研究：为什么四步循环能改变一次任务的结果

中心问题：为什么同一个模型，把任务跑成 Plan→Patch→Verify→Learn 就稳，一把梭就反复失败？
从表层现象追到机制、失败模式、权衡与边界。标注编辑综合。

## 表层现象

- 现象 A：一句「帮我做好」下去，Agent 直接大改，方向错了才发现，回退成本高。
- 现象 B：一次 diff 同时改了实现、顺手重构、还动了别的文件，审查者看不过来，只能盲信。
- 现象 C：Agent 说「已完成」，但没有任何客观信号，跑起来才发现是坏的。
- 现象 D：这次踩的坑，下次换个任务又原样踩一遍，因为教训只留在聊天记录里。

## 机制一：先规划能把错误挡在落盘之前（Plan）

Claude Code plan mode 的行为很直白：只读文件、提出计划、批准前不落盘。
它把「决定改什么」和「动手改」分开。分开的价值是：方向性错误在最便宜的时候（还没写代码）被发现。
计划里最该写清的不是长篇方案，而是四个会真正改变结果的约束——文件范围、禁区、最快检查、回滚方式。
这也是为什么本文 Showcase 的 plan.md 只有几行：它不是文档，是一次任务的护栏。

## 机制二：小切片让 diff 可被真正审查（Patch）

「切片小」不是美学偏好，而是可审查性的前提。一个实用判据：你能在 5 分钟内读完这次 `git diff`。
git diff 是逐行差异，人脑一次只能可靠地审查有限的变化；一旦一次 diff 里藏了多个决定，
审查就退化成「看起来没问题」。所以规则是：一次 patch 只承载一个决定。
本文 Showcase 把它做实：切片一只做「均分」这一个动作（+3/−2 行），故意不处理余数，
让下一步的验收去暴露缺口——而不是在同一次改动里既均分又补余数，把两个决定糊在一起。

## 机制三：验收要用客观信号，而不是模型自述（Verify）

AGENTS.md 明确：列出的程序化检查，Agent 会尝试执行并在完成前修复失败。
关键在「可执行」：命令能真正运行并产出退出码，形容词不能。
所以 Verify 的核心不是问「你完成了吗」，而是跑一条最快的相关检查看退出码。
本文 Showcase 用 `node --test`：切片一退出码 1，断言直接打印 `sum(3333,3333,3333) 应为 10000，实际 9999`——
这不是模型的自我评价，是环境给的事实。切片二修好后退出码 0。退出码把「完成」变成可判定的。

## 机制四：把教训显式写回，形成可审计的跨会话规则（Learn）

CLAUDE.md 文档的限定句是这一步的支点："Claude treats them as context, not enforced configuration."
Claude Code 也有 auto memory，会根据用户纠正自动记笔记；因此不能声称显式文件是唯一跨会话路径。
本文仍要求把关键教训写回 AGENTS.md/CLAUDE.md，因为显式规则能进版本控制、接受审查并留下 before/after 证据；
留在聊天里又没有被显式或自动沉淀的经验，不能假定下次一定会出现。
本文 Showcase 把「金额拆分先写求和守恒断言、余数单独切一片」写回 AGENTS.md（before/after 有 diff 为证）。
下次再做同类任务，这条规则会作为上下文被读到——这就是循环闭合的地方。

## 失败模式与对应步骤（编辑综合）

| 缺失步骤 | 触发的失败 | 一手依据 |
| --- | --- | --- |
| Plan | 无计划直接大改、方向错了才发现、范围漂移 | plan mode：只读提计划、批准前不落盘 |
| Patch | 一次 diff 藏多个决定、审查退化成盲信 | git diff 逐行差异；可审查性有上限 |
| Verify | 只信「已完成」、坏结果混过验收 | AGENTS.md：可执行程序化检查、退出码 |
| Learn | 同类坑反复踩、关键教训没有形成可审计规则 | CLAUDE.md / auto memory：跨会话上下文有显式与自动两条路径 |

## 权衡

- 步子太重：给一个改一行的小任务也写长计划、写复盘，成本高于收益。
- 步子太轻：省掉 Verify 直接信模型，短期快，坏结果的发现被推迟到运行时甚至线上。
- 折中：按任务风险和不可逆性缩放。低风险改动 Plan 可以只有一句、Learn 可以不写；
  高风险或反复出现的任务，四步都要，且 Verify 与 Learn 不能省。

## 边界条件（什么时候这套不适用或不够）

- 一次性、零风险的小实验（试个想法就删），写四步的成本可能高于收益。
- 无法定义「成功是什么」的需求：连验收都写不出来时，先补任务定义，而不是硬跑循环。
- 循环只保证「有护栏、有信号、有回写」，不保证内容正确：Verify 的检查本身写错，照样放过坏结果；
  高风险不可逆动作（发布、删云数据、发消息）必须在循环之外叠加人工审批与工具强制（hook/权限/沙箱）。
- 四步是操作节奏，不是审计口径；系统性审计用五组件模型（另文）。

## 编辑综合结论

Plan→Patch→Verify→Learn 不是新概念的堆叠，而是把四项已存在的一手能力，
按「一次任务」的时间顺序串成闭环，并给每步一个客观锚点。
它把失败从运行时提前暴露，把一次性的教训沉淀成可复用的规则。这是 LearnPrompt 的操作模型，不是行业标准。
