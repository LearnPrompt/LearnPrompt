# 纵向研究：为什么“冻结一个可验收切片”是 MVP 的真正最小单位

从表层现象一路追到机制、失败模式和边界。带“【编辑综合】”的段落是本文的操作性
判断，非某个来源的直接结论。核验日期 2026-07-11。

## 表层现象

把“我想做个 AI 新闻雷达”直接交给 coding agent，常见结果是：它要么问一堆问题
卡住，要么自作主张做了一个又大又跑不通的东西——抓取、界面、登录全糊在一起，
最后你无法判断它到底做完没有。

## 往下一层：Agent 需要的不是灵感，是一个可判断的契约

Anthropic 在长任务 harness 实践里给出的关键动作是：**在写代码前先约定 done 长什么样**，
并用带硬阈值的验收标准来判定每一轮是否通过，还要把“做事的”和“判断做对的”分开。
把这个要求前移到需求阶段就得到一个推论——

【编辑综合】交给 Agent 的最小单位不是“一个想法”，而是“一个能被验收的切片”。
只要验收动作还写不出来，需求就还没冻结，不管提示词写多长都一样。

## 再往下：把范围削减拆成四个可回答的维度

模糊需求之所以无法执行，是因为四件事没定。逐一定死，需求就从开放创作变成确定任务：

1. **输入**：数据从哪来、长什么样。没有输入契约，Agent 只能猜。
2. **输出**：交付物的确切形态（一份 Markdown？一个 JSON？一个页面？）。
3. **失败态**：输入缺失、非法、零结果时怎么办。不定义失败，就没有边界。
4. **验收动作**：跑哪条命令、看到什么算成功。这是把“完成”变成退出码的关键。

【编辑综合】这四维不是我发明的分类，而是把 harness 的“能力/约束/状态/验收”
要求收敛到需求阶段的最小提问集。它的作用是让“砍到多小”有一个可操作的停止条件：
四件事都能写死，就够小了；任一件还含糊，就还不能开工。

## 失败模式

- **把 MVP 写成商业计划**：同时要登录、支付、推荐、后台、动画。对照 MVP 原意，
  这恰恰违背“最小投入”，但更致命的是它没有一个能一次实现的切片。
- **只定义成功路径，不定义失败态**：Agent 在脏数据上崩溃或静默产出空结果，
  你却以为它“做完了”。Showcase 里“零命中必须非零退出”就是在防这个。
- **验收靠肉眼**：没有 `verify` 脚本，就只能反复目测，无法接进自动化，也无法
  让第二个角色独立判断。
- **backlog 隐形**：好点子不写进 backlog，就会以“顺便加一下”的方式回流进首版，
  把切片重新撑大。

## 边界与适用条件

- 本方法针对**还没有仓库、还没有任务**的模糊想法。一旦切片冻结、进入已有仓库，
  接力棒应交给 project-checklist 的开工清单和 Plan→Patch→Verify→Learn 循环。
- 它不替代真实的商业验证。冻结一个能跑的切片，只证明“技术上第一版能交付并验收”，
  不证明“市场需要它”。
- 对确实无法先本地跑通的需求（强依赖外部账号、实时数据、硬件），最小切片可以先
  用 fixture 或桩替身把主流程跑通，把真实接入留在 backlog——这正是 Showcase 的做法。

## 结论

【编辑综合】MVP 的“最小”不是砍功能的艺术，而是**冻结的纪律**：把用户价值收敛到
一个主流程，把输入 / 输出 / 失败态 / 验收动作写死，其余推进 backlog。四件事写清那一刻，
需求才第一次变成 coding agent 能一次做完、你能客观验收的东西。
