# Brief：AI 编程项目的开工与验收清单

## 目标读者

刚开始用编码 Agent（Claude Code、Codex 等）接手一个已有仓库的初学者。
读者会写提示词，但还没养成“开工前先写清单”的习惯。

## 问题

同一个模型，交给它清晰的项目上下文就能稳定改代码，丢给它一句“帮我改一下”就反复失败。
多数失败不是模型不够聪明，而是开工时没有把仓库、运行方式、禁区、密钥、验收、回滚写清楚。
旧稿只有 41 行、一份抽象清单，没有机制、没有可复现证据、还在渲染 SourceCard。

## 中心主张

把项目开工清单从散文变成一份六格、可机器校验的契约，能把“到底写没写清楚”
从主观判断变成确定性闸门；这份清单一次写好、每个任务复用。

## 学习目标（读完能做什么）

1. 用六个必答问题写出一份可复用的开工清单（可落成 AGENTS.md / CLAUDE.md）。
2. 理解每一格对应哪种失败，以及它映射到 Harness 的哪个组件。
3. 用一段脚本把清单校验成退出码，接入开工前检查。

## 非目标

- 不重复相邻文章：工具选型见《Claude Code 与 Codex 怎么选》；四步循环见《Agentic Coding 的最小工作流》；五组件抽象模型见《Harness 的五个组件》。本文只聚焦“开工清单这份具体工件”。
- 不做模型跑分，不给工具排名。
- 不覆盖 CI/CD、团队流程等更大范围。

## 需要的证据

- AGENTS.md 标准、CLAUDE.md 记忆、Claude Code 权限（deny 规则）、git 回滚的一手资料。
- 一个可复现 Showcase：把清单校验成退出码，并演示缺项被拦截。
- 一张原创教学图：六格清单映射到失败与 Harness 组件。

## 验收条件

- `cd starlight && npm run build` 退出码 0。
- 确定性校验脚本 `node scripts/validate-golden-mdx.mjs` 通过。
- Showcase 真实运行、脱敏冻结。
- 保持 `showcase_status: partial`，不自评 quality_score，等独立审稿。

## 已记录的假设（无人值守下的最小安全假设）

- 保留 legacy `source_repo`/`source_url` 指向 harness-engineering-orange-book，仅作迁移元数据与主题地图；产品与工程事实回到官方一手资料。
- 六格字段沿用旧稿骨架（仓库分支、运行检查、禁改目录、密钥、验收、回滚），因为它们与一手资料推荐的内容一致。
