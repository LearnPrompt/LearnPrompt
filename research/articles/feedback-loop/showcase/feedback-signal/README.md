# Showcase：模糊反馈 vs 可操作反馈

这个 Showcase 用一个最小仓库回答一个问题：同一次失败，反馈信号的颗粒度不同，下一轮能不能定位到最小改动。它保留一份可复现的 fail → 诊断 → 最小 patch → pass 全过程，并把两条反馈通道并排放在一起。

它证明的不是某个模型更聪明，而是：只有当反馈信号带上文件、行号、期望值和实际值时，一次失败才能被诊断成一处最小改动；只回“失败了”三个字的信号，会逼着人重写或瞎猜。

## 目录

```text
feedback-signal/
├── repo/
│   ├── src/slugify.mjs         # 任务对象（当前保存为失败状态：漏掉小写化）
│   └── test/slugify.test.mjs   # 验收标准：小写、连字符、去标点三条用例
├── checkers/
│   └── vague-check.mjs         # 模糊反馈通道：只回 PASS/FAIL，吞掉定位信息
├── patch/
│   └── slugify.mjs             # 最小改动版本（trim 后加一步 toLowerCase）
└── result.txt                  # 冻结的真实运行输出（已脱敏）
```

可操作反馈通道不需要额外脚本：`node --test` 自带文件、行号、期望值和实际值。模糊反馈通道 `vague-check.mjs` 跑同一份测试，却只把退出码翻译成一句“构建未通过”，故意丢掉全部定位信息，用来代表“全站构建只说失败了”这类粗颗粒信号。

## 环境

- macOS，Node v24.11.0，无网络依赖。
- 所有命令都在本目录内以相对路径运行。

## 复现步骤

```bash
# A. 模糊反馈：只拿到一个布尔式结论
node checkers/vague-check.mjs               # 期望 FAIL，退出码 1，无定位信息

# B. 可操作反馈：同一次失败，颗粒度完全不同
node --test repo/test/slugify.test.mjs      # 期望 3 条挂 1 条，给出 file:line + expected/actual

# C. 从证据定位最小改动，应用后重跑同一命令
cp patch/slugify.mjs repo/src/slugify.mjs
node --test repo/test/slugify.test.mjs      # 期望 3 条全过，退出码 0
```

## 2026-07-11 的真实结果

- 模糊通道：`vague-check.mjs` 只回 `FAIL: 构建未通过（无更多信息）`，退出码 1。
- 可操作通道：`node --test` 三条里挂一条，指出 `repo/test/slugify.test.mjs:5`，期望 `hello-world`、实际 `Hello-World`，把搜索范围收窄到大小写行为；内层用例耗时不到 1ms，整个 runner 约 39ms。
- 候选修复：失败信号把范围收窄到小写化；本文选择在 `trim()` 后补一步 `.toLowerCase()`，只改一行。重跑同一命令后三条全过，但这不证明它是唯一实现。

完整脱敏输出见 `result.txt`。

## 它证明了什么

- 反馈的价值不在“报没报错”，而在信号是否带定位：文件、行号、期望值、实际值。
- 同一次失败，可操作反馈能把搜索范围收窄到小写化相关修复；模糊反馈只给一个布尔结论，逼人重写或瞎猜。
- “修好”不是靠感觉，而是重跑同一条验收命令、退出码从 1 变 0。

## 它没有证明什么

- 两种反馈通道跑的是同一份测试和同一次失败，差别只在信号颗粒度，不代表任何模型能力对比；本 Showcase 未运行在线大模型，因此不构成任何模型排名。
- 一次 pass 不等于真实任务长期可靠；单元测试只覆盖了三条用例，真实验收还要靠更完整的测试、构建与人审。
- 检查器只判断信号里有没有定位信息，不判断人是否会用它；拿到 file:line 仍可能改错地方。
