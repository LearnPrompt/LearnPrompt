# 纵向研究：一个 Skill 为什么会“不被触发”或“到处乱触发”

中心问题：读者写完第一个 `SKILL.md`，最常见的失败不是内容差，而是触发行为不对。
把这个问题从现象追到机制、失败模式和边界。

## 现象层

- 写好了 Skill，Agent 却当没看见，仍然用普通提示词硬做。
- 或者相反：一个“写作 Skill”在几乎所有任务里都被拉起来，污染无关对话。

## 机制层（一手：官方文档三级加载）

关键在于 Level 1 元数据始终加载、且**只有 `name` 和 `description` 进入系统提示**。
也就是说，Agent 在决定“要不要用这个 Skill”时，先看到这两项，正文（Level 2）
要等请求匹配后才用 bash 读进来。官方进一步用“请求匹配 Skill 的 `description`”描述读取条件，因此可以得出：

- `description` 是主要匹配依据，必须描述清楚“做什么 + 什么时候用”。正文写得再好，
  也无法补救模糊的发现元数据，因为匹配时正文还没加载。
- `description` 太宽（“需要处理文本时使用”）→ 命中面过大 → 到处乱触发。
- `description` 太窄或只写“做什么”不写“何时用”→ 该触发时不触发。

官方对 `pdf-processing` 的示例 `description` 同时写了能力（extract text and tables、fill forms、
merge documents）和触发场景（when working with PDF files or when the user mentions PDFs...），
正是这个道理。

## 字段边界层（一手：官方文档 + 可运行脚本）

触发器不是自由文本，有硬边界，写错会在打包前被挡下：

- `name`：≤64 字符、仅小写字母/数字/连字符、不含 XML 标签、不含保留词 `anthropic`/`claude`。
- `description`：非空、≤1024 字符、不含 XML 标签。

Showcase 用官方 `quick_validate.py` 亲手复现了三种被拒写法（大写空格的 name、
多余字段、缺 description），证明这些边界是机器强制的，不是风格建议。需要注意：当前脚本
会检查 `description` 的尖括号，并用 kebab-case 正则排除 `name` 中的尖括号，但未单独检查保留词；权威边界仍以官方文档为准。

## 渐进披露如何反过来指导结构（编辑综合）

既然正文（Level 2）建议低于 5k tokens、脚本与资源（Level 3）按需加载且代码不进上下文，
那么“把长脚本塞进 SKILL.md 正文”既浪费上下文又难维护。更好的做法是：

- SKILL.md 正文只留核心工作流与判断；
- 稳定、需要确定性的操作放 `scripts/`，让 Agent 经 bash 执行、只吃输出；
- 大段参考资料放 `references/`，用到才读。

这条推论是本文对官方“progressive disclosure”原则的操作化综合，用于回答“目录该怎么分层”。

## 失败模式与边界

- 只写理念不写输入输出：Agent 读完仍不知何时触发、怎么执行。
- 触发场景描述技术偏好而非任务（“需要抓数据时”）：命中面过宽。
- 验收只写“高质量”：没有可检查条目，等于没验收。
- 安全边界缺失：涉及账号、密钥、发布、删除时容易出事；官方也强调 Skill 的安全审计。
- 什么时候**不该**写 Skill：一次性任务、需求仍在探索、没有稳定步骤且高度依赖人工判断——
  这些场景普通提示词更划算，做成 Skill 只会增加维护负担。

## 小结

第一个 Skill 写不好，常见根因不是文笔，而是发现元数据和字段边界没处理对，
以及没有按三级加载把内容分层。把这三件事做对，Skill 才能被稳定触发、执行和验收。
