# Brief：从自然语言需求到可运行 MVP

## 要解决的问题

很多人把一句产品想法直接丢给 coding agent：“我想做一个 AI 新闻雷达。”
Agent 于是自行脑补用户、数据源、算法、界面和完成标准，产出一个庞杂、跑不通、
也无法验收的东西。问题不在模型不够聪明，而在需求还停在“商业愿景”阶段，
没有被冻结成一个边界清楚、能一次实现、能客观验收的最小切片。本文要把这个
冻结过程讲成可操作的动作，并用一次真实的最小 MVP 证明它。

## 目标读者

想用 coding agent 把某个产品想法快速做出第一版的实践者：会用命令行、能读
JSON 和一小段脚本，但常常败在“需求太大、交不出可验收的第一版”。

## 学习目标

读者读完能够：

1. 把一句模糊想法改写成一条可执行的用户价值（谁 / 在什么场景 / 得到什么结果）。
2. 用四个维度削减范围：输入、输出、失败态、验收动作。
3. 写出一张能交给 Agent 一次实现的冻结任务卡，并显式记录 backlog 边界。
4. 用一段验收脚本把“怎么算完成”变成退出码，而不是靠肉眼判断。
5. 识别“再补一句提示词”补不了的范围问题。

## 中心结论

MVP 的最小单位不是“砍功能”，而是**冻结一个能被验收的切片**：把用户价值收敛到
一个主流程，把输入 / 输出 / 失败态 / 验收动作四件事写死，其余全部推进 backlog。
四件事一旦写清，实现就从“开放式创作”变成“确定性任务”，coding agent 才能一次做完。

## 明确不做

- 不讲商业 MVP、市场验证、增长这些产品经理话题；只讲交给 Agent 前的技术冻结。
- 不重复相邻文章：开工六格清单见 project-checklist，Plan→Patch→Verify→Learn
  循环见 minimum-agentic-coding-workflow，审批边界见 plan-auto-approval-boundary。
- 不用一次本地实验给任何工具或模型排名。
- 不复制橙皮书或第三方图片；橙皮书仅作中文主题地图链接。

## 需要的证明

- 一次可复现的真实过程：从一句自然语言需求出发，保留澄清与冻结后的任务卡，
  实现一个很小但能跑、能验收的 MVP，保存命令、输出、失败态与局限。
- 一张原创教学图，讲清“模糊需求 → 四维冻结 → 最小切片 / backlog”的机制。
