# Workflow comparison

| 观察项 | Claude Code：现场探索 | Codex：完整委派 |
| --- | --- | --- |
| 初始输入 | “把短清单改得更实用”，先不写 | 已冻结的目标、结构、边界、验收 |
| 首个有价值产物 | 三个内容缺口、四个编辑问题 | 可审查的单文件 diff |
| 人工介入 | 回答一次方向问题，并处理执行故障 | 执行中没有澄清 |
| 执行故障 | 会话悬挂、插件工具冲突、turn 上限、依赖缓存路径 | 有全局插件警告，但没有阻断任务 |
| 修改范围 | 1 个 MDX，`+96/-18` | 1 个 MDX，`+192/-18` |
| 验收 | 外层 Harness 在修正依赖环境后运行 | Agent 在同一次委派中运行 |
| 最终构建 | PASS，48 pages | PASS，48 pages |

## 这次真正证明了什么

- 当需求仍是“改得更实用”时，先进入交互式 discovery 有价值，因为模板、受众、案例和课程分工都会改变文章。
- 当这些选择已经冻结，任务就可以被压缩成一个异步 handoff；执行者不需要在中途等待编辑回复。
- 工具选择不能脱离 Harness。Claude 路线的几次失败来自全局插件、会话连续性和依赖路径，而不是文章难度。
- 两边都能写出并构建一篇教程。本实验不评价文字优劣，也不证明某个模型普遍更快。

## 选型句

需求还在形成、你愿意持续在场：选择交互式路线。
任务卡已冻结、你只想回来审查 diff：选择完整委派路线。
