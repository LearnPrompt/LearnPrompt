# vertical-research.md — 向下追机制、权衡与失败模式

验证日期：**2026-07-11**

## 中心问题

怎样把一句“用 Claude Code 做个 Chrome 扩展原型”压缩成一个**可加载、可验证、权限克制**的 Manifest V3 contract？

## 一、先冻结验收，而不是先问 Claude 写哪些文件

如果任务只写成：

> 帮我做一个网页信息收集扩展。

Claude Code 很容易合理地“过度完成”：

- 加上常驻 `host_permissions`；
- 顺手申请 `tabs` 或 `<all_urls>`；
- 先塞后台脚本、选项页、图标、同步存储；
- 甚至接入远程 API 或账号体系。

这些都不一定是错，但它们会让你失去最宝贵的两个东西：

1. **你不知道最小成功条件是什么。**
2. **你不知道多申请的权限是不是必要。**

所以本文的 contract 必须先冻结成：

- 从当前页面读取 **标题、URL、可选选中文本**；
- 保存到 `chrome.storage.local`；
- popup 显示最近一次保存结果；
- 只申请必要权限；
- 有 fixture、README、确定性检查、失败或越权负例；
- 能在 Chrome for Testing 150 中完成机械 22-check，并把 branded Chrome 150 的 `Load unpacked` 人工验收边界写清楚。

这一步不是文书工作，而是架构工作。因为一旦验收标准这样写，很多结构选择会自动收窄。

## 二、为什么主扩展只该申请 `activeTab`、`scripting`、`storage`

### `activeTab`

官方文档说明，`activeTab` 只在用户调用扩展时，给当前页一个临时访问窗口，并且**不触发权限 warning**。这正符合“阅读卡”这种显式、一次性的操作：用户按一次快捷键，扩展才读取当前页。

它的好处是：

- 不需要长期 host 权限。
- 不需要 `<all_urls>` 这种大范围授权。
- 读者更容易理解“权限和动作一一对应”。

它的坏处也要说清楚：

- 权限不是常驻的，离开当前页或切到受限页面就会失效。
- 这类设计更适合“用户触发一次、抓一次”，不适合后台自动扫页。

### `scripting`

我们需要从页面上下文里拿 `document.title`、`location.href` 和 `window.getSelection()`。这不是 popup 自己的 DOM，而是当前网页的 DOM。MV3 的标准做法就是 `chrome.scripting.executeScript()`。

它的含义不是“可以做任何注入”，而是：

- 你可以在用户授权过的当前页执行一小段打包在扩展内的代码；
- 扩展逻辑仍留在包内，符合 MV3 对远程托管代码的限制。

### `storage`

popup 关闭后会消失，service worker 也不是常驻背景页。如果不落到 `chrome.storage.local`，最近一次抓取结果就没有稳定归宿。官方文档同时提醒：扩展应该优先用 `chrome.storage`，而不是把持久状态塞进网页 `localStorage`。

### 为什么不申请 `tabs`

Chrome `tabs` 文档说明，`tabs` 权限的关键价值，是读取 `tabs.Tab` 上的敏感属性，例如 `title` 和 `url`。但本文故意不用这一层：

- tab 的 `id` 和“当前活动 tab”查询，本例不需要额外敏感权限；
- 真正要展示给读者看的标题与 URL，直接从页面上下文读取；
- 这样能把“为什么不用 `tabs`”讲得更清楚。

### 为什么不申请 `host_permissions`

如果目标是“当前页阅读卡”，那就不需要常驻权限。常驻 host 权限意味着：

- 你可以在匹配域名上自动运行脚本；
- 同时也扩大了出错半径；
- 还让教程更像“生产插件脚手架”，而不是“最小原型”。

编辑综合判断：**只有当需求升级为自动抓取、跨页批量处理、或无需用户触发就得运行时，才值得重新讨论 host 权限。**

## 三、为什么“快捷键触发 + service worker 抓取 + popup 只读展示”比静态 content script 更适合样稿

可以想象另一种实现：在 manifest 里声明静态 content script，让它在每个页面都运行，然后把结果发回扩展。

这条路并非不可行，但不适合作为这篇黄金教程的中心 showcase，原因有三：

1. **权限含义会变复杂。** 静态 content script 常常伴随 match patterns 或 host 权限，读者更难一眼看出“它到底能读哪些页面”。
2. **验收边界不如显式动作清晰。** 快捷键触发一次，抓一次，权限与操作直接对应；常驻脚本则更像一条长期驻留的机制。
3. **失败模式不够收敛。** 当前页阅读卡要讲的是“从验收标准反推最小结构”，不是“扩展平台所有注入方式的比较综述”。

因此本文选择：

- 快捷键负责显式触发；
- service worker + `executeScript()` 负责一次性读取页面状态；
- `storage.local` 负责保存最近一次结果与最近一次运行状态；
- popup 只读展示，不再和权限入口缠在一起。

这正好形成一条读者能在脑中画出的链路。

## 四、为什么还要保留最小 service worker

按功能说，这个样稿可以勉强把逻辑继续塞进 popup，但保留一个最小 service worker 有两个现实好处：

1. **更贴近 MV3 的真实世界。** 官方资料明确指出，MV3 已从长期 background page 转向按需唤醒的 service worker。教程如果完全跳过这一层，读者会误以为“扩展背景执行模型不重要”。
2. **更利于机械验证。** 自动化时可以通过 service worker target 确认扩展已经真实加载，而不是只看目录结构。

这里的设计关键不在于“让 service worker 负责更多逻辑”，而在于**克制**：只负责监听快捷键、抓当前页和写入状态，不做与本题无关的后台流程。

## 五、失败模式必须进入 contract，否则 Claude 很容易把它们藏掉

这题至少有两类失败，必须在教程里正面写出来。

### 1. 受限页面失败

当当前页是 `chrome://...` 这类受限页面时，`executeScript()` 不应假装成功。真正好的样稿要把这个失败变成：

- popup 中的明确错误文案；
- showcase 里的冻结失败输出；
- 文章里的使用边界说明。

否则读者上线后第一次碰到受限页，就会把“权限不够”误判成“代码坏了”。

### 2. 越权 manifest

另一个常见问题不是“运行时报错”，而是**一开始就申请过头**。例如：

- `tabs`
- `unlimitedStorage`
- `<all_urls>`

这类 manifest 可能照样能跑，但它违背了本文的中心判断：**教程应该先证明最小权限已足够。**

所以越权不是“以后再优化”的小毛病，而应该作为负例进入 deterministic check。

## 六、Claude Code 在这题上的真正作用

这篇不是要证明 Claude Code 神奇地懂 Chrome 扩展，而是要说明它最适合承担哪一段工作。

最适合 Claude Code 的部分：

- 根据冻结 contract 生成最小文件结构；
- 把页面提取、popup 呈现和本地存储拼成一致的数据流；
- 按照现有约束修 manifest 和 UI 文案；
- 跑本地检查并回传证据。

最不该交给 Claude Code“自由发挥”的部分：

- 需求边界本身；
- 权限是否过大；
- 什么算成功；
- 哪些失败必须保留为证据。

编辑综合判断：

> 在浏览器扩展题目里，Claude Code 最有价值的不是替你“想创意”，而是当你已经写清验收后，快速把那个 contract 变成一套最小、可跑、可查错的源文件。

## 七、写作综合结论

本文要交付给读者的，不该只是一个能跑的 demo，而应该是一套可迁移的方法：

1. 先写验收，不先堆权限。
2. 先做最小动作，不先做全能插件。
3. 先保留失败证据，不先追求“看起来顺利”。
4. 先用 Chrome for Testing 做机械验证，再把 branded Chrome 的人工边界说清楚。

这也是本文最终文章结构的依据：先讲 contract，再讲结构，再讲真实 showcase，再讲越权与失败模式，最后讲边界和练习。
