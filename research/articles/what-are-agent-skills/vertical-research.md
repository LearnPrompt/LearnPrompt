# Vertical Research：什么时候值得从 prompt 升级成 Skill

核验日期：2026-07-12。

## 真实问题

“我已经有一个常用 prompt，要不要做成 Skill？” 读者真正难的不是不会写 `SKILL.md`，而是不知道**什么时候这件事已经从一次性意图，变成了值得沉淀的工作包**。

这不是文风问题，而是四层边界问题：

1. 这次用户到底只是在表达一次意图，还是已经在反复要求同一套流程？
2. 这套流程里哪些部分需要长期保留为自然语言判断，哪些部分应该下沉成确定性脚本？
3. 哪些约束应该始终生效，哪些应该按需加载？
4. 这套东西未来是给自己复用，还是还要分发给别人？

## 编辑部综合：从 prompt 升级成 Skill 的四个门槛

以下四条是 LearnPrompt 编辑部操作模型，不是官方术语：

### 1. 重复发生

如果一个请求只出现一次，就让 prompt 完成。Skill 的维护成本包括目录、说明、测试、版本和路径约束，只有在同类任务会再次出现时才划算。

### 2. 边界稳定

你至少已经知道：

- 输入长什么样；
- 输出要长什么样；
- 哪些错误该直接停下；
- 什么叫“完成”。

如果这些还在变，做 Skill 往往只是把模糊性固化起来。

### 3. 复用价值超过提示词长度

“prompt 很长”本身不是升级理由。真正的升级信号是：同一套说明里既有触发元数据、又有固定步骤、又有参考资料、又有验收边界，继续把它塞进聊天框会让 discoverability 和维护都变差。

### 4. 机械部分已经能下沉

如果规则里有一段天然应该确定性执行，例如文件名归一化、schema 校验、patch gate、privacy scan，那么 Skill 的价值就更明确：正文保留人类判断，脚本承担机械转换。

## Skill 的加载链

### Prompt 解决“这次想做什么”

prompt 是单次意图。它回答的是“这次要做哪件事”，而不是“以后遇到类似任务默认按什么流程做”。

### Skill metadata 解决“何时该唤起这个工作包”

开放规范和 OpenAI / Claude 当前文档都把 `name` / `description` 放在 discovery 入口。元数据决定这个工作包能否被发现、是否值得被显式调用。

### Skill body 解决“进入后该怎么协调”

正文不是脚本替身，而是工作手册：输入、步骤、失败边界、验收。

### references / scripts / assets 解决“细节放哪一层才不浪费上下文”

- `references/`：长说明或政策文件，只有需要时再读。
- `scripts/`：确定性转换。
- `assets/`：模板、schema、示例。

这也是本文教学图要表达的加载链。

## 为什么普通脚本还不够

单独一个脚本只能回答“怎么做变换”，回答不了：

- 什么时候该运行这段脚本；
- 需要先读哪份命名政策；
- exit `21` 和 `23` 各是什么意思；
- 这次只允许 dry run 还是允许真实 rename。

也就是说，脚本负责**转换**，Skill 负责**协调与解释**。两者缺一不可，但不能互相冒充。

## receipt-renamer 案例为什么值得做成 Skill

这组任务已经符合四个门槛：

1. 它是重复发生的导入清理任务。
2. 输入稳定：一批 receipt manifest 和源 PDF。
3. 输出稳定：dry-run JSON 计划和 Markdown 摘要。
4. 机械部分稳定：日期、金额、slug、冲突检查都可确定性执行。

真正值得沉淀的不是“把文件名改好”这一个动作，而是整套工作包：

- 先确认 dry run；
- 读取命名政策；
- 运行确定性脚本；
- 根据 exit code 解释失败；
- 可选再跑测试；
- 向人类汇报计划而不是直接改文件。

这已经明显超出了“一次性 prompt”的边界。

## 边界表：谁负责什么

### Prompt

负责这次意图和上下文，比如“请先处理 normal-batch，再跑测试”。它不应该承担长期 discoverability。

### Skill

负责按需复用的工作流，把元数据、正文、参考和脚本组织成一个可被发现和显式调用的包。

### `AGENTS.md` / `CLAUDE.md`

负责始终适用的项目约束，例如允许路径、验证命令、代码风格和 review 规则。它们不该塞进每个单独 Skill 的正文里重复出现。

### 普通脚本

负责确定性转换和机械检查。它不解决 discovery，也不负责解释“何时使用”。

### MCP

负责跨出本地工作区，去读外部系统、数据或工具。它不该承担本地流程说明。

### plugin

负责把 Skill、MCP、hooks 等一起打包分发。它不该替代 Skill 的内部工作流，也不该替代 MCP 的连接层。

## 什么时候不要做 Skill

1. 这件事只做一次。
2. 规则还在探索，每做一次都要改一遍流程。
3. 只有一个短脚本和一个固定命令就够，根本不需要额外的发现元数据或解释层。
4. 真正缺的是外部系统接入，那应该先上 MCP，而不是强行写一个“会读外部世界”的 Skill。

## 编辑部综合与风险

### 综合

“值不值得 Skill 化”不是看 prompt 长短，而是看它是否已经长成一个**可发现、可复用、可解释、可验收**的工作包。

### 风险

- 很多人会过早 Skill 化，把仍在探索的模糊流程硬封装起来。
- 也有人只保留脚本，不保留工作流说明，导致下一个人只能读代码猜用法。
- 还有人把 plugin 当逻辑层、把 MCP 当工作流层，最后维护成本飙升。
