# Vertical Research — content-automation-pipeline

## 中心问题

怎样把一个听上去像“帮我抓一下、总结一下、最好顺手发掉”的一次性任务，重构成 Claude Code 能安全承接的七阶段内容流水线？

## 先拆开四种最容易混淆的状态

### 1. 没抓到内容

这是真正的 `no_content`。它要求：

- 快照本身存在；
- 字段完整；
- 去重和打分都跑过；
- 最终确实没有任何条目满足入选规则。

### 2. 抓到了数据，但数据坏了

例如来源条目缺 `source_url`。这时你不能对编辑说“今天没有内容”，因为事实上有内容，只是不能安全进入草稿。正确状态应该是 `blocked_data_quality`，并把错误定位到具体来源、具体记录和具体字段。

### 3. 草稿已经生成，但草稿 contract 被下游改坏

这正是本轮新增的 `verify-failed`。输入快照、去重和打分都正确，但草稿在 `verify` 之前被可控篡改，例如某条 `source_url` 被改成错误值。正确状态不是“草稿差不多能看”，而是 `failed_contract`，因为草稿已经不再忠实反映被冻结的入选结果。

### 4. 草稿已经生成，但还不能向外走

这是本文最重要的状态边界。`draft` 只是“本地草稿已成形”，`publish-candidate` 才是“经过人工批准，可以进入发布候选池”。二者之间至少隔着一个明确的批准工件，不能靠一句 prompt 假定“请谨慎发布”。

## 为什么要先冻结 snapshot，而不是边抓边总结

一旦你边抓边总结，就会同时失去三样东西：

1. **可重放性**：下一位编辑不知道你到底看到了什么输入。
2. **可归因性**：你不知道错误发生在抓取、清洗、出稿还是验证。
3. **可讨论性**：团队无法围绕同一份输入讨论是否该入选。

因此本文把 `snapshot` 放在第一层，而且只承认冻结后的离线 JSON。以后即使要接入实时抓取，也只能新增“写入快照”的上游步骤，而不能让抓取直接改写后续阶段的输入。

## `normalize` 不是“顺手清洗”，而是第一道真实性闸门

绝大多数一次性脚本会把清洗写成一小段映射：有就保留，没有就 `undefined`。这在内容生产里非常危险。少了 `source_url` 之后，Agent 仍可能继续生成一段看上去像周报的文案，但读者、编辑和审稿人都无法追溯每条结论来自哪里。

所以 `normalize` 必须承担两个责任：

1. 把来源字段统一成固定 contract。
2. 在 contract 不成立时明确失败，而不是自动降级成空结果。

Showcase 的 `missing-source-field` 负例正是围绕这件事设计的：第二份快照有一条 `openai-codex-cloud` 缺 `source_url`，结果停在 `normalize`，退出码 `21`，并写明 `no_content: false`。这条 `false` 比错误信息本身还重要，因为它阻止了系统把“数据坏了”说成“今天没东西”。

## `dedupe` 和 `score` 解决的是“为什么选它”，不是“模型觉得像”

如果你只让模型读两份源然后出一篇周报，最终很难回答“为什么这条被选中、那条没进来”。`dedupe` 和 `score` 的价值正好在这里：

- `dedupe` 解决“同一条消息在两个源里都出现”时如何保留一条主记录；
- `score` 解决“哪些条目值得进入本周草稿”时如何留下可检查的理由。

本文 Showcase 用的是极朴素的确定性规则：

- 命中 `claude-code` 或 `codex` 主题；
- 直接涉及 workflow、hooks、permissions、verification；
- 来源是官方文档；
- 人工重要度高。

这些规则不追求“最聪明”，追求的是“下周别人接手时能看懂为什么是这 4 条”。如果以后要接更复杂的排序模型，也应该把模型输出写回一个单独工件，而不是让它直接改写最终周报。

## `draft`、`verify`、`approve` 是三个不同阶段，不可合并计数

很多内容自动化脚本会把“生成 Markdown 文件”直接视作完成。问题在于，生成完成、通过 contract 校验、可以进入发布候选，不是同一件事。

### `draft`

`draft` 的职责是把入选条目转成可编辑草稿，同时保留：

- `source_url`
- 来源快照
- 发布时间
- 筛选理由
- 摘要

### `verify`

`verify` 的职责是机械检查草稿是否仍然忠于前序阶段：

- 每条入选记录的 `source_url` 是否还在；
- 筛选理由是否还在；
- 去重后是否又重新出现重复 `source_url`；
- 草稿中的关键信息是否与 `score.manifest.json` 对得上。

这一步的意义，在于拦截“输入没坏、打分没坏，但草稿被后处理改坏”的场景。`verify-failed` 正是这个反例：`draft` 阶段故意把 `openai-codex-cli` 的 `source_url` 改成错误值，结果 `verify` 真实退出 `23`，并在 `verify.manifest.json` 写下 `draft missing source_url for openai-codex-cli`。

### `approve`

`approve` 的职责不是再做一次事实校验，而是做状态提升：

1. **编辑空间**：草稿允许人工删改标题、顺序和表述。
2. **责任边界**：只要还没批准，系统就不能自称“已发布”。
3. **审计边界**：任何对外动作都应该有单独工件，而不是与生成动作混在一起。

因此 Showcase 的成功路径虽然会生成 `publish-candidate/*.md`，但 manifest 仍写明 `external_publish_triggered: false`。这不是啰嗦，而是刻意把“进入发布候选”和“已经外发”拆成两个完全不同的状态。

## 把 Claude Code 官方边界映射到这条流水线

### 官方 workflow 资料告诉我们的，不是术语，而是工作方法

`Common workflows` 页面给出的三个点与本文直接相关：

- 用 worktrees 运行隔离会话，避免并行编辑互相碰撞；
- 用 `plan` 模式做到“先审后改”；
- 把 Claude 接进脚本和批处理，而不是把所有逻辑都塞进一次交互里。

这正好对应本文的三个操作习惯：

1. 研究文章和跑 showcase 可以在隔离目录内独立完成；
2. 文章改写本身适合先冻结 plan，再落磁盘；
3. 周报流水线更适合沉淀成零密钥脚本和固定 fixtures，而不是临时聊天。

### Hook 资料告诉我们的，是“什么动作必须确定发生”

官方 `hooks-guide` 和 `hooks` reference 强调：Hook 是生命周期上的确定性控制层，`PreToolUse` 可对一次工具调用做 deny / ask / allow 判断。这对本文的启发是：

- 如果将来你真的接入“推送到 CMS”“调用发布 API”之类动作，不应该只靠 prompt 说“请谨慎发布”；
- 你应该把那种高风险动作接到显式门禁上，必要时让 Hook 或权限系统把它挡住。

本文 Showcase 没有真实发布步骤，但保留了这个边界：`approve` 只复制到本地 `publish-candidate/`，没有任何网络动作。

### 权限资料告诉我们的，是“真正的裁判是谁”

官方 `permissions` 页面把规则说得很清楚：allow / ask / deny 由 Claude Code 权限系统执行，prompt 或 `CLAUDE.md` 并不能改变底层允许什么。这意味着：

- “人工批准后才进入发布候选”不能只写成一句文字要求；
- 它至少要落成一份可检查的批准工件，必要时再叠加 Hook 或权限规则；
- 如果你以后接了真实发布工具，那一步的 allow / ask / deny 必须是系统级门禁，而不是文案约束。

## 为什么说“模型摘要只是可替换边界”

本题最容易被误解的地方，是以为“模型摘要”就是整条流水线的核心。其实不是。

真正稳定的骨架是：

`snapshot -> normalize -> dedupe -> score -> draft -> verify -> approve`

模型最多只该出现在 `score` 之后、`draft` 之前，承担“把若干入选条目写得更像可读摘要”的工作。即使未来把 deterministic 模板换成 Claude Code、别的模型，甚至人工润色，也不能改变下面三件事：

1. 事实输入必须仍来自冻结快照。
2. 每条输出仍要保留 `source_url` 与筛选理由。
3. 没有批准工件就不能进入发布候选。

换句话说，模型是中间件，不是裁判，也不是发布按钮。

## 失败模式与代价

### 代价

- 文件数增加：要维护 fixtures、manifest、draft、candidate、visual check。
- 编写成本更高：第一次做会比“一把梭生成一篇周报”慢很多。
- 规则需要维护：打分和字段 contract 以后还要迭代。

### 收益

- 下一位编辑可以接着干，而不是从头猜。
- 失败原因可定位，不会把数据坏了说成没内容。
- 草稿篡改会在 `verify` 被拦住，不会混进待发布候选。
- 发布边界清晰，默认只产草稿，不会顺手外发。

### 不适合的场景

- 你只需要私下看一眼两份材料，不打算复用。
- 来源极其不稳定，连固定 contract 都还没想清楚。
- 团队没有人愿意看 manifest，也没有人工批准环节。

这套方法的意义不在“更自动”，而在“更可接手”。
