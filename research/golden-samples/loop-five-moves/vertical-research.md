# 纵向研究：五个动作如何形成真正的闭环

验证日期：2026-07-10

## 1. Discover：不是遍历所有输入

发现层要把“世界上所有可能的事情”压缩成有边界的候选队列，并去重、排序、过滤过期项。示例把它简化成 `tasks/*.json` 中第一个未完成任务。生产场景还需要 watermark、消息 ID 或时间窗。

## 2. Handoff：传任务，不传模糊对话

下游 worker 至少需要目标、输入、验收条件和允许的资源。示例写出 `handoff.json`，这样即使 worker 更换模型或进程，任务定义仍能独立读取。

## 3. Verify：闭环里最难的一步

“生成了一段文本”不是完成。验证应尽量落到可机械判断的条件，并与 worker 分离。示例比较实际文本与 `expected_text`；真实内容任务可能使用 schema、事实核对、规则检查、抽样或人工 review。无法定义验收的任务，不应过早自动循环。

## 4. Persist：保存可继续的状态

状态不是一篇长日志。最小状态应包含已完成项、运行次数、最后任务和最后结果；详细输入输出放在每次 run artifact 中。这样下一次既能快速决定，也能追溯证据。

## 5. Schedule：根据结果决定下一步

调度不是单纯 cron。它至少要区分 `continue`、`retry` 和 `idle`，并对失败设置退避、预算与人工升级。示例中成功且无剩余任务就进入 idle；验证失败则返回 retry。

## 6. 什么时候不要做 Loop

- 一次性探索，输入与完成标准都还在变化。
- 高风险外部动作，没有撤销、审批或人工验收。
- 验证成本高于任务本身。
- 错误会在轮次间累积，却没有可靠的去重与状态恢复。

先把一次执行做成可验证 Harness，再考虑让它自动发现下一次工作。
