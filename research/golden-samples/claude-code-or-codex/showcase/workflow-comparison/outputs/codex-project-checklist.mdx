## 这份清单是干什么的

很多初学者接手已有仓库时，第一反应是“先让 AI 跑起来看看”。问题是，只要项目边界没写清楚，Agent 很容易在错误目录里改文件、误碰凭据、或者把“已生成草稿”当成“已发布”。

项目清单的作用不是写文档好看，而是把下面几件事提前钉死：

- 这次到底在哪个仓库、哪个目录、哪个阶段工作。
- 哪些文件和动作可以改，哪些绝对不能碰。
- 完成以后用什么真实信号验收，而不是靠口头判断。
- 如果中途换人，下一位能不能无缝接着做。

如果你还没看过完整闭环，可以先读 [Agentic Coding 的最小工作流](./minimum-agentic-coding-workflow)；这篇专门解决“接手旧仓库时，第一张项目卡该怎么写”。

## 五分钟开工表

刚接手仓库时，不要先改代码，先把下面这张表填出来。目标不是全面，而是五分钟内把最关键的不确定性压掉。

| 问题 | 去哪里找 | 至少留下什么 |
| --- | --- | --- |
| 这是哪个仓库、这次在哪个子目录工作？ | README、项目根目录、文档站目录说明 | 仓库名、工作目录，例如 `starlight/` |
| 这次唯一确认过的检查命令是什么？ | 现有文档、任务说明、已知命令 | `cd starlight && npm run build` |
| 哪些目录是本次只读盘点，不能顺手清理？ | 迁移说明、历史目录、任务边界 | “旧站目录只读，不在本轮删除” |
| 需要什么凭据，谁来提供？ | 环境说明、部署说明、团队约定 | “凭据由人工注入，Agent 不写入仓库” |
| 什么算完成，什么还不算？ | 验收口径、发布流程、负责人说明 | “构建通过” 和 “已发布” 分开写 |
| 中途换人时，下一位最需要看到什么？ | 你的任务卡、变更记录 | 当前状态、已跑检查、未决风险 |

你可以把这张表理解成“开工前的最低上下文”。没有它，后面每一步都会反复打断你问同样的问题。

## 可复制 YAML 项目卡

下面这张卡适合直接贴到 `AGENTS.md`、任务文档、Issue 描述，或者任何交接页里。它不是追求完整，而是保证 Agent 和下一位接手者都知道边界。

```yaml
project: LearnPrompt Starlight migration
repo: learnprompt-showcase
workdir: starlight/
task: 把旧文档迁移成 Starlight 页面，并保持可构建
goal:
  - 只修改本次指定文档
  - 保留现有 frontmatter、URL 语义和 SourceCard
  - 输出可扫读、可执行的中文教程
do_not_touch:
  - 旧站历史文件，仅盘点，不在本轮删除
  - 凭据、发布配置、线上内容
acceptance:
  - 运行 `cd starlight && npm run build`
  - 构建通过，无新增 MDX 语法错误
handoff:
  - 记录改动文件
  - 记录当前结构
  - 记录构建结果
  - 记录剩余不确定性
secrets:
  - 如需凭据，由人工通过环境变量或平台注入
publish_boundary:
  - 构建通过不等于已发布
  - 发布动作必须人工确认
```

## 没有清单时为什么一定返工

很多返工不是因为模型差，而是因为任务卡太短。下面这个对比，基本就是接手旧仓库时最常见的前后区别。

### Before

```md
帮我把这个仓库整理一下，顺便把文档迁过去，最后确认能不能上线。
```

这句话的问题在于：

- “这个仓库”没有说清楚工作目录。
- “整理一下”默认鼓励顺手重构。
- “迁过去”没有写旧目录是否可删。
- “上线”把构建、预览、发布混成一个动作。

结果通常是：Agent 一边迁内容，一边清理历史目录，还可能开始碰部署相关配置。最后你需要花更多时间把边界补回来。

### After

```md
只修改 `starlight/src/content/docs/ai-coding/project-checklist.mdx`。
保留现有 frontmatter、URL 与 `SourceCard`。
旧 Docusaurus 内容只读盘点，不在本轮删除。
完成后只运行 `cd starlight && npm run build` 作为验收。
构建通过即可交付，不执行任何发布动作。
```

这时候任务就从“泛化的大改造”收缩成“单页迁移 + 单条验收命令”。审查成本、返工概率、交接难度都会明显下降。

:::caution[凭据和发布动作]
接手旧仓库时，最危险的不是改错一段文案，而是误碰凭据和发布边界。

- 不要把 token、密钥、`.env` 内容抄进任务卡、文档或提交记录。
- 不要把“构建通过”写成“已经发布”。
- 不要默认让 Agent 执行发布、上线、对外发送、权限修改这类动作。
- 如果任务确实涉及凭据，应该写明“由人工通过环境变量或平台注入”，而不是让 Agent 自行猜测。
:::

## 实操步骤

接手已有仓库时，可以按下面顺序工作：

1. 先写项目卡，而不是先开改。
2. 把“可改范围”和“只读范围”分开写。
3. 把“构建”“预览”“发布”拆成不同状态。
4. 每做完一轮改动，就补一行 handoff：改了什么、验证了什么、还剩什么风险。

一个够用的最小手写版清单通常只要这几项：

- 工作目录。
- 允许修改的文件范围。
- 不允许碰的目录、配置、账号或动作。
- 唯一确认过的验收命令。
- 交付时必须回报的状态字段。

## LearnPrompt Starlight 迁移完整案例

下面用一个真实的迁移边界来说明，为什么项目清单不是可选项。

### 场景

目标是把 LearnPrompt 的一篇 AI 编程教程迁移到 `starlight/` 文档站里，并且保留已有来源信息组件。旧站内容仍然存在，但这一轮不做清理。

### 如果没有清单

常见失控路径是这样的：

1. Agent 看到“迁移文档站”，开始同时读新站和旧站。
2. 发现旧内容风格不统一，顺手改标题、删历史页面、补新组件。
3. 看到可能有发布链路，进一步猜测是否还要处理预览或线上站点。
4. 最后即使文档内容没问题，你也说不清这轮到底改了什么、哪些目录被碰过、是否还能安全交给下一位。

### 有清单时怎么写

把边界写成下面这样，就会稳定很多：

```yaml
project: LearnPrompt Starlight migration
workdir: starlight/
edit_scope:
  - starlight/src/content/docs/ai-coding/project-checklist.mdx
preserve:
  - existing frontmatter
  - existing source_url
  - existing SourceCard
read_only:
  - 旧 Docusaurus 内容，仅用于盘点
verification:
  - cd starlight && npm run build
done_when:
  - 文章结构完整
  - 相对链接可读
  - 构建通过
not_in_this_round:
  - 删除旧内容
  - 修改发布配置
  - 执行发布动作
```

### 这张卡具体帮你挡住了什么

- 它把工作目录压到 `starlight/`，避免 Agent 在仓库别处游走。
- 它把修改范围压到单文件，避免顺手重构。
- 它要求保留 `SourceCard` 和来源 URL，避免迁移时丢出处。
- 它把旧站定义成“只读盘点”，避免误删历史资产。
- 它把验收固定为 `cd starlight && npm run build`，避免编造命令。
- 它明确“本轮不发布”，所以构建通过以后就该停。

这就是一个完整、可执行、可交接的迁移案例。它并不复杂，但足够把错误率降下来。

:::note[新项目差异]
如果你做的是全新项目，而不是接手旧仓库，这张清单要改两处：

- “只读历史目录”通常不存在，重点会变成初始化目录结构和首个验收命令。
- “迁移边界”会变成“首版范围边界”，例如只做一个页面、一个接口、一个脚本，而不是一次把整站搭完。
:::

## 练习

试着给你手上的任意一个旧仓库写一张 10 分钟版本的项目卡，至少回答下面四题：

1. 这次唯一允许改的目录是什么？
2. 哪个目录或动作是只读、禁止顺手处理的？
3. 你当前唯一确认过的验收命令是什么？
4. 你会怎样把“构建通过”和“已发布”分开记录？

如果你回答不出来，先别开工，继续补项目卡。

## 读后验收清单

看完这篇以后，你应该能独立完成下面这些事：

- 能在五分钟内写出一张接手旧仓库的开工表。
- 能复制 YAML 项目卡，并替换成自己项目的边界。
- 能解释为什么没有清单会造成返工。
- 能把凭据和发布动作单独列成高风险边界。
- 能把一个真实迁移任务压缩成“工作目录 + 修改范围 + 验收命令 + 交付状态”。
- 能在交接时明确写出“已构建、未发布、仍有风险”的状态。
