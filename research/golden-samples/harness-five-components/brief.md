# Brief：Harness 的五个组件

## 要解决的问题

很多人把 Agent 失败归结为模型不够强，却没有检查指令、工具、权限、状态和验收是否完整。本文要给出一个可以直接审计的五组件框架，并用最小目录证明每个组件都能落到文件和检查上。

## 目标读者

已经能调用 coding agent 或模型 API，准备让 Agent 处理真实仓库、文档或自动化任务的实践者。

## 学习目标

读者能够：

1. 区分模型能力与 Harness 提供的运行条件。
2. 用五个问题检查一个 Agent 系统是否完整。
3. 运行最小 Harness showcase，并知道它与生产安全的差距。
4. 识别“再加一段提示词”无法解决的工程问题。

## 中心结论

“Harness 的五个组件”不是行业唯一分类，而是 LearnPrompt 的操作模型：指令、能力、约束、状态、编排。它们分别回答 Agent 应做什么、能做什么、不能做什么、记住什么，以及谁负责推进和验收。

## 明确不做

- 不把五组件宣称为统一标准。
- 不把示例配置宣称为生产级沙箱。
- 不用更多规则掩盖缺失的测试和反馈。

## 验收条件

- 交叉引用 OpenAI 与 Anthropic 的一手工程资料。
- Showcase 的五个组件都能被确定性脚本验证。
- 正文包含失败模式、简化原则和升级路径。
- 独立审稿与站点构建通过。
