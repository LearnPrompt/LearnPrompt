# 纵向研究：从“指令没用”一路追到机制与边界

围绕一个关键问题向下追：为什么写了指令，Agent 还是跑偏？编辑综合部分已显式标注。核验日期 2026-07-11。

## 现象层：指令写了却不管用

三种最常见的失败：不读规则就改、范围越改越大、自己宣布完成。表面看像“模型不听话”，很多人的第一反应是再补一句“请认真、保持高质量”。但补形容词几乎不改变结果。

## 机制层一：指令是上下文，不是强制配置

Claude Code 官方明说 CLAUDE.md 被当作上下文而非强制配置，要真正拦住一个动作得用 PreToolUse 钩子。这解释了为什么“不要碰密钥”写进指令仍可能被跨过：指令只影响模型的选择概率，不构成硬闸。由此得到第一条分工——指令层解释意图，约束层执行拒绝。

## 机制层二：可执行 = 每个维度落到真实资源

官方对“有效指令”的建议是具体、简洁、结构化，并给了对照例子：“Run `npm test` before committing” 优于 “Test your changes”。把这条推到底，指令层其实要回答五个能被定位的问题：

- 目标：做成什么样，且可判断（不是“更好”而是“返回 YYYY-MM-DD”）。
- 范围：能动哪些文件（具体路径，不是“相关文件”）。
- 顺序：先读什么、再做什么、最后验什么。
- 验收：用什么命令判定（`node --test` 全过，不是“我觉得做完了”）。
- 冲突优先级：维度打架时听谁的。

Showcase 的确定性检查器就是把这五问机械化：模糊指令 0/5，可执行指令 5/5。这不是文字游戏——0/5 意味着没有任何一个维度能落到可运行的资源上。

## 机制层三：冲突不是措辞问题，是优先级缺失

官方两条一手事实合起来解释了“跑偏”的根因：CLAUDE.md 里“两条规则矛盾时模型可能任选其一”，Codex 里文件按根到叶拼接、越近越优先且有 32 KiB 上限。也就是说，冲突在真实工程里必然发生（多层 CLAUDE.md、全局与项目 AGENTS.md、模糊目标与隐藏验收），而系统默认的解法要么是“任选其一”，要么是“越近越赢”。指令层要做的是把这件事从默认行为改成显式声明。

Showcase 把它演成一个最小冲突：“让日期更好读”（人类可读串）与测试规定的机器格式 `2026-07-11` 直接对撞。模糊指令下二者无人裁决，候选实现顺着“更好读”写出 `Sat, 11 Jul 2026`，验收 3 挂 2；补上“冲突时以测试为准”后，同一验收 3 条全过。

## 机制层四：优先级阶梯

把冲突优先级再拆细，得到一条稳定阶梯（编辑综合，基于官方分工与安全常识）：

1. 安全与范围约束（最高）：不越界、不碰密钥、不发布——且由系统强制。
2. 显式任务指令：本轮目标与验收，压过通用偏好；对应 agents.md 的“explicit user chat prompts override everything”。
3. 通用风格偏好（最低）：好听但会被上层验收否决。

同层冲突才升级给人。这条阶梯让“指令冲突”从玄学变成可执行规则。

## 边界层：指令层解决不了什么

- 解决不了强制：拒绝、沙箱、审批属于约束层，用钩子/权限实现。
- 解决不了工具缺失：那属于能力层，补形容词无用。
- 解决不了跨轮记忆：目标、上次失败证据属于状态层。
- 解决不了谁来验收、失败回到哪一步：属于编排层。
- 指令本身也有物理上限：CLAUDE.md 越长遵循度越低、建议 200 行内；Codex 合并项目指令达到 `project_doc_max_bytes`（默认 32 KiB）后停止继续加入后续文件，不是把已合并内容整体丢弃。因此“把所有知识塞进入口文件”本身就是反模式，应留导航、把细节下沉到就近文档、rules 或 skill。

## 编辑综合（非事实，需标注）

- “五个可执行维度”是 LearnPrompt 对官方“具体、可验证”建议的操作化归纳，不是官方术语。
- “优先级阶梯（安全 > 显式任务 > 通用偏好）”是基于官方分工与安全常识的编辑判断，agents.md 只直接支撑其中“显式 prompt 覆盖一切”一条。
- 两个候选实现由作者手写代表模糊/精确指令的导向；可复现的确定部分是验收 harness 与其真实输出，不构成模型排名。
