# Brief：反馈循环（feedback-loop）

## 目标文章
- 路径：`starlight/src/content/docs/agent-engineering/feedback-loop.mdx`
- 章节顺序：agent-engineering / order 4（约束层之后、记忆层之前）
- 类型：Agent 工程深度教程（重写旧占位稿）

## 问题
Agent 失败后，很多人只把“再认真一点”写回提示词，或只粘贴最后一行报错。真正决定下一轮能否修好的，是反馈信号的质量：它有没有把失败变成一个可定位、可判断、低延迟、可执行的证据。旧占位稿只有四条操作步骤，没有讲清机制、延迟、最小改动定位和停止/升级条件。

## 目标读者
已经在用 Claude Code / Codex 等 Agent、遇到过“改了半天越改越乱”的工程师与进阶用户。

## 一句话产出
读完能用四个属性审计任何一条反馈信号，把最快验收放进最内圈，从失败证据定位一处最小改动，并说清何时停止、何时升级。

## 中心主张
反馈层的工作不是“报没报错”，而是把环境信号变成下一轮可执行的最小修正；信号带上文件、行号、期望值与实际值时，一次失败才能被诊断成一处最小改动，模糊信号只会逼人瞎猜或大改。

## 非目标
- 不做模型能力排名（Showcase 两条通道跑同一份测试，只比信号颗粒度）。
- 不替代 evaluation（评测）专题；本文只讲反馈层如何用评测产出的验收信号。
- 不深入 CI/沙箱实现细节，那属于约束层与编排层。

## 需要的证据
- 一手来源支撑“给可验证信号”“分离生成与评审”“停止/升级”等工程主张。
- 一个可复现 Showcase：fail → 诊断 → 最小 patch → pass，且并排展示模糊反馈与可操作反馈。
- 至少一张教学图，解释反馈回路 + 信号质量 + 停止/升级。
